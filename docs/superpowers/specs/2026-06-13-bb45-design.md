# BB45 — Design Spec

**Date:** 2026-06-13
**Status:** Approved (brainstorming complete)

## What it is

"BB45" — a first-person "what I'd see looking out from my deck" ship watcher for a
house in Blue Bay, Curaçao. NOT a top-down map (that's MarineTraffic's job). Ships
are drawn against the real horizon at their true bearing and apparent size, framed
by the deck edge, with a weather-driven visibility edge. Personal project, one
house. Simple and fun.

The screen is a faithful mirror of what the eye actually sees from the deck — you
look up at a ship far out, look down at the screen, and it's in the same spot.
Realism comes from two physically-honest channels (bearing and apparent size); the
only stylized axis is a gentle vertical nudge for legibility.

## Stack & conventions

- Vanilla JS + HTML Canvas frontend. No framework, no bundler.
- Native ES modules (no build step). Served by a tiny zero-dep static server.
- Node 20+ and the `ws` package for the AIS relay (milestone 3 onward only).
- Units metric (km, m, knots, °C). Timezone America/Curaçao (AST).
- No secrets in the frontend. No localStorage. Near-zero dependencies.
- Clean, minimal-comment, 2-space-indent style.

## Decisions (deltas from original brief)

1. **Vertical placement = gentle/honest.** Bearing→x and distance→apparent-size are
   exact physics. The only stylized axis is vertical, governed by one tunable
   `EXAGGERATION ≈ 0.3` constant calibrated against the real view (milestone 6).
   An apparent-size cap keeps a close cruise ship "big" without filling the screen.
2. **Sightline = physics-backed (Koschmieder) + calibration knobs**, surfaced as a
   question, not an assertion. Three states: **hidden** / **"barely — can you see
   it?"** (faint ghost ridge) / **clearly visible**. A future yes/no answer is
   reserved as a calibration signal that nudges the model constants.
3. **Split ES modules + tiny zero-dep static server** (`npm start`, Node 20+).

## The geometry (the heart — drives everything)

- **Horizontal (exact):** `Δ = normalizeSigned(bearingTo(deck,ship) − viewBearing)`;
  cull if `|Δ| > fov/2`; else `x = W·(0.5 + Δ/fov)`.
- **Apparent size (exact, capped):**
  `apparentWidthPx = (shipLength_m / distance_m) · (W / fovRadians)`, capped at a
  fraction of screen width.
- **Vertical (stylized, tunable, honest at the limits):** the horizon line is the
  truth; nearer ships drop by `EXAGGERATION · nearness`. Bearing and size never
  move with this.
- **Hull-down (kept physically exact):** `horizonKm(h) = 3.57·√h`. Visible to the
  waterline within `horizonKm(deckHeight) ≈ 18.9 km`; hull progressively clipped
  beyond that; only superstructure visible out to
  `horizonKm(deck) + horizonKm(superstructure) ≈ 38 km`; culled past that. Most
  mid-range ships sit hull-down on the horizon — real, and the reason the gentle
  vertical nudge matters for legibility. Tooltip notes "hull-down — superstructure
  only" when applicable.
- **Render gate:** `distance ≤ min(geometricHorizon, weatherSightline)`, fading
  into haze near the limit. Venezuela's 900 m peaks clear the geometric horizon at
  70 km easily, so the land ridge is gated by aerosol sightline, while ships are
  mostly gated by the ~38 km superstructure horizon.

## The scene (visual core)

- Sky gradient + soft sun + drifting clouds; sea gradient with a sun-glitter
  streak; a gently curved horizon (slight Earth curvature) for depth.
- Deck foreground, minimal so nothing interrupts the sea: a shallow band of
  white/cream travertine at the very bottom (subtle perspective seams), then a
  frameless, fully transparent glass edge — no posts, no top rail, just a faint
  glass tint/reflection so water reads straight through. A palm tree in each
  bottom corner (trunk from off-frame, fronds arcing inward), simple dark
  semi-transparent silhouettes intruding ~15% into each corner.
- A thin compass/bearing strip along the bottom edge of the sea (above the stone
  band), with degree + cardinal ticks.
- Ships as simple type-specific silhouettes (container, tanker, bulk, cruise,
  coaster, yacht), faded by atmospheric perspective with distance.
- Hover a ship → tooltip: name, flag, type, destination, distance (km), bearing
  (deg + 16-pt compass), speed (kn), length; "hull-down — superstructure only"
  when applicable.
- Panels: live weather (top-left) and a "how far you can see" panel with the
  Venezuela verdict + a draggable Sightline (km) slider, a drift play/pause, and a
  "live wx" toggle.

## Deck constants (source of truth; currently ESTIMATES, calibrated in m6)

`lat 12.1349, lon -68.9853, height 28 m, viewBearing 202°, fov 108°.`
Landfall: Venezuela, bearing ~196°, distance ~70 km, peaks ~900 m.

## Simulated ships (behind USE_SIM)

~9 plausible vessels `{id,name,flag,type,dest,course,kn,lat,lon}` within the view
arc at 4–45 km, drifting along course each frame (accelerated so motion is
visible), recycling to the opposite side when they leave the arc or pass ~55 km.

## Data sources

- **Ships (live, m3+):** aisstream.io, `wss://stream.aisstream.io/v0/stream`.
  Subscribe with JSON `{ Apikey, BoundingBoxes, FilterMessageTypes }`.
  BoundingBoxes = `[[[swLat,swLon],[neLat,neLon]]]` hugging the coast, e.g.
  `[[[12.02,-69.12],[12.20,-68.84]]]`. Use PositionReport (lat/lon/Sog/Cog) and
  ShipStaticData (Destination, Type, Name); fields under `m.Message.<Type>` and
  `m.MetaData`. GOTCHA: aisstream drops idle/slow connections (~2 min) — the relay
  MUST auto-reconnect with backoff and RE-SEND the subscription on every (re)open.
- **Weather + clarity (browser-direct, no relay):** Open-Meteo. Forecast fields
  temperature_2m, relative_humidity_2m, wind_speed_10m, wind_direction_10m,
  weather_code, cloud_cover, visibility; plus air-quality API for dust and
  aerosol_optical_depth. GOTCHA: `visibility` caps ~24 km and is near-surface only
  — it cannot answer the 70 km Venezuela question. That's an AEROSOL question;
  June is Saharan dust season. Derive long-range sightline from AOD + humidity
  (Koschmieder) and gate the Venezuela ridge on it.

## Relay contract (m3+)

Relay listens on `ws://localhost:8080`. Relay → browser:
`{ type:"ship", mmsi, lat, lon, sog, cog, name?, dest?, shipType? }` and optionally
`{ type:"status", connected:bool }`. Frontend keeps a `Map<mmsi,ship>` and rebuilds
`ships[]`. Map AIS numeric ship-type codes → silhouette buckets; derive flag emoji
from the MMSI MID (first 3 digits).

## Milestones (in order, each independently shippable)

1. Scaffold + CLAUDE.md + git. Delete the Java starter; create package.json
   (type:module), public/ and server/ folders, empty server/.env plus
   server/.env.example; .gitignore covering node_modules/, server/.env, .idea/,
   mockups/, .superpowers/. git init + first commit.
2. Build the visual core: full scene + geometry + live weather + simulated ships
   (USE_SIM=true), served by `npm start`. **STOP here for browser review.**
3. Relay: server/relay.js → aisstream with key from .env, subscribe with bbox,
   forward normalized ship messages, reconnect logic.
4. Frontend wiring: connect to relay, feed ships[] from live data, USE_SIM off.
5. Ship identity: AIS type-code → silhouette; MMSI MID → country flag.
6. Calibration: replace DECK estimates with measured values; wire the Venezuela
   yes/no answer into the sightline constants.

## Out of scope today

Auth, accounts, databases, deployment. Local-only.

## Risks tracked

- Sightline constants are estimates until calibrated — model is structurally right
  and improves with observation.
- Canvas hover hit-testing is hand-rolled (store each ship's screen rect per frame).
- DECK estimates mean bearings/sizes are approximate until m6.
